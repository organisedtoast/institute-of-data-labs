Relational Databases 101 - Blog

This page teaches you relational database fundamentals using a blog as a real-world example. You'll learn how tables, keys, and relationships work together to organise data.

PK Primary Key FK Foreign Key UNIQUE No duplicates allowed COMPOSITE Two fields act together

What You Are Looking At

A logical data model is a picture of how information is organised. Each box is a table, each row inside a box is a field, and the lines between boxes show how tables are related.

How to read one table Start with the table name, then read the fields underneath it. For example, the Post table stores details about blog posts such as the title and the user who wrote it.
How to spot keys A field marked PK uniquely identifies a row. A field marked FK points to a row in another table.
How to read relationships If one table connects to many rows in another table, that is a one-to-many relationship. Example: one user can create many posts.

Key Concepts

PK Primary Key A primary key is the field that uniquely identifies each row in a table. No two rows in the same table can share the same primary key value.
FK Foreign Key A foreign key stores a value that points to the primary key in another table. This is what links tables together.
UNIQUE Unique Attribute A unique field does not have to be the primary key, but its value still cannot be repeated. In this model, two users cannot share the same email.
COMPOSITE Composite Key A composite key means two fields work together as the unique identifier for a row. In the Like table, user_id and post_id together identify one like.

Relationships in Plain English

  • One User can author many Post rows.
  • Each Post belongs to one User through Post.user_id.
  • One User can write many Comment rows.
  • One Post can have many Comment rows.
  • One User can give many Like rows.
  • One Post can receive many Like rows.

Worked Example

Here is one small story that matches the model exactly. Following a real example makes the diagram easier to understand.

Step 1
Alice signs up.

A row is created in User for Alice.

Step 2
Alice writes a blog post.

A row is created in Post and its user_id points back to Alice.

Step 3
Bob leaves a comment.

A row is created in Comment with both a post_id and a user_id.

Step 4
Bob likes the post.

A row is created in Like using Bob's user_id and the post's post_id.

Why this story matters One simple scenario shows how all four tables connect: users create posts, users write comments, and users like posts. The model is not just four separate boxes; it is one connected system.

Primary Key vs Foreign Key

Beginners often confuse these because both are important identifiers, but they do different jobs.

Primary Key

  • Uniquely identifies one row in its own table.
  • Example: User.user_id identifies one specific user.
  • Think: "Who is this row?"

Foreign Key

  • Points to a primary key in another table.
  • Example: Post.user_id points to User.user_id.
  • Think: "Which other row is this connected to?"

Composite Key in the Like Table

Why does Like use two key fields? A like is not just about the user and not just about the post. It is about the specific combination of one user liking one post. That is why user_id and post_id together form the key.

If Bob has user_id = 2 and Alice's post has post_id = 101, then the pair (2, 101) identifies one specific like.

What Problem Would Happen Without This?

  • Without a primary key, it would be hard to uniquely identify a specific row.
  • Without a foreign key, a comment could point to a post that does not exist.
  • Without UNIQUE on email, two users could register with the same email address.
  • Without the composite key in Like, one user could like the same post again and again as duplicate rows.

From Diagram to Database

Entities become tables The boxes named User, Post, Comment, and Like become tables in a database.
Attributes become columns Fields such as username, title, and created_at become columns inside those tables.
Keys become constraints Primary keys, foreign keys, and unique rules are turned into database constraints so the data stays valid.
Relationships become links Columns like Post.user_id and Comment.post_id are what let SQL join related data together.

Common Beginner Mistakes

  • Mixing up Post.post_id with Post.user_id. One identifies the post; the other points to the author.
  • Thinking a comment belongs only to a user. It belongs to both a user and a post.
  • Not understanding why Like is a separate table. It exists because many users can like many posts.
  • Assuming UNIQUE means the same thing as PK. They are related, but not identical.

Quick Quiz

Question 1 (Tables & Fields)

Which table stores who wrote a comment?

Answer: Comment, using user_id.

Question 2 (Tables & Fields)

What does the created_at field store?

Answer: A timestamp showing when the row was created.

Question 3 (Tables & Fields)

Which table would you query to find all comments on a specific post?

Answer: The Comment table, filtered by post_id.

Question 4 (Primary Keys & Uniqueness)

Can two users have the same user_id?

Answer: No - user_id is a primary key, so each value must be unique.

Question 5 (Primary Keys & Uniqueness)

Why is email marked UNIQUE?

Answer: So two users cannot share the same email address.

Question 6 (Foreign Keys)

What field links a post back to its author?

Answer: Post.user_id.

Question 7 (Foreign Keys)

The Comment table has two foreign keys. What are they and what do they point to?

Answer: post_id points to Post (which post), and user_id points to User (who wrote it).

Question 8 (Relationships)

What type of relationship exists between User and Post?

Answer: One-to-many (one user can author many posts).

Question 9 (Composite Keys)

Why does Like need both user_id and post_id?

Answer: Because a like is defined by one user liking one post.

Question 10 (Composite Keys)

Why is the Like table separate instead of being a field in the Post table?

Answer: Because many users can like many posts - a separate table handles this many-to-many relationship.

Question 11 (Data Integrity)

If Alice deletes her account, what happens to her posts?

Answer: They would become orphaned (point to a non-existent user) - this is why foreign key constraints matter.

Question 12 (Queries & Joins)

If you wanted to display all posts by a specific author, which two tables would you join?

Answer: User and Post, joined on User.user_id = Post.user_id.
