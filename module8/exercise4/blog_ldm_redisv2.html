Blog Data Model and Redis Basics

This page teaches two ideas in order. First, it explains the blog's relational data model using SQL tables, keys, and relationships. Then it shows how Redis can sit beside that database as a fast cache layer for repeated reads.

PK Primary Key FK Foreign Key UNIQUE No duplicates allowed COMPOSITE Two fields act together Redis comes after the ERD

What You Are Looking At

A logical data model is a picture of how information is organised inside the main relational database. In the ERD below, each box is a table, each row inside a box is a field, and the lines between boxes show table relationships. Redis is important too, but Redis is not a relational table, so we will teach it separately after the ERD.

How to read one table Start with the table name, then read the fields underneath it. For example, the Post table stores details about blog posts such as the title and the user who wrote it.
How to spot keys A field marked PK uniquely identifies a row. A field marked FK points to a row in another table.
How to read relationships If one table connects to many rows in another table, that is a one-to-many relationship. Example: one user can create many posts.

Key Concepts

PK Primary Key A primary key is the field that uniquely identifies each row in a table. No two rows in the same table can share the same primary key value.
FK Foreign Key A foreign key stores a value that points to the primary key in another table. This is what links tables together.
UNIQUE Unique Attribute A unique field does not have to be the primary key, but its value still cannot be repeated. In this model, two users cannot share the same email.
COMPOSITE Composite Key A composite key means two fields work together as the unique identifier for a row. In the Like table, user_id and post_id together identify one like.

The Entity Relationship Diagram ('ERD')

This ERD shows only the relational data model, because an ERD is for permanent entities, columns, and relationships in the main database.

How to read this ERD correctly Start with the core path User → Post → Comment. Then notice that Like is a separate table because many users can like many posts. Redis is not shown here on purpose because it is a cache layer, not part of the relational schema.

Relationships in Plain English

  • One User can author many Post rows.
  • Each Post belongs to one User through Post.user_id.
  • One User can write many Comment rows.
  • One Post can have many Comment rows.
  • One User can give many Like rows.
  • One Post can receive many Like rows.
  • The Like table exists because this is a many-to-many relationship, not a simple field on Post.
  • All of these relationships belong to the SQL database, which remains the source of truth for the blog data.

Worked Example

Here is one small story that matches the model exactly. Following a real example makes the diagram easier to understand.

Step 1
Alice signs up.

A row is created in User for Alice.

Step 2
Alice writes a blog post.

A row is created in Post and its user_id points back to Alice.

Step 3
Bob leaves a comment.

A row is created in Comment with both a post_id and a user_id.

Step 4
Bob likes the post.

A row is created in Like using Bob's user_id and the post's post_id.

Why this story matters One simple scenario shows how all four tables connect: users create posts, users write comments, and users like posts. The model is not just four separate boxes; it is one connected system.

Primary Key vs Foreign Key

Beginners often confuse these because both are important identifiers, but they do different jobs.

Primary Key

  • Uniquely identifies one row in its own table.
  • Example: User.user_id identifies one specific user.
  • Think: "Who is this row?"

Foreign Key

  • Points to a primary key in another table.
  • Example: Post.user_id points to User.user_id.
  • Think: "Which other row is this connected to?"

Composite Key in the Like Table

Why does Like use two key fields? A like is not just about the user and not just about the post. It is about the specific combination of one user liking one post. That is why user_id and post_id together form the key.

If Bob has user_id = 2 and Alice's post has post_id = 101, then the pair (2, 101) identifies one specific like.

What Problem Would Happen Without This?

  • Without a primary key, it would be hard to uniquely identify a specific row.
  • Without a foreign key, a comment could point to a post that does not exist.
  • Without UNIQUE on email, two users could register with the same email address.
  • Without the composite key in Like, one user could like the same post again and again as duplicate rows.

From Diagram to Database

Entities become tables The boxes named User, Post, Comment, and Like become tables in a database.
Attributes become columns Fields such as username, title, and created_at become columns inside those tables.
Keys become constraints Primary keys, foreign keys, and unique rules are turned into database constraints so the data stays valid.
Relationships become links Columns like Post.user_id and Comment.post_id are what let SQL join related data together.

Common Beginner Mistakes

  • Mixing up Post.post_id with Post.user_id. One identifies the post; the other points to the author.
  • Thinking a comment belongs only to a user. It belongs to both a user and a post.
  • Not understanding why Like is a separate table. It exists because many users can like many posts.
  • Assuming UNIQUE means the same thing as PK. They are related, but not identical.
  • Trying to draw Redis as if it were another SQL table. Redis can store useful copies of results, but it is not part of the relational ERD.
Redis Performance Layer

Redis Primer for a Complete Beginner

The ERD answered the question "where does the real blog data live?" Redis answers a different question: "how can the app return repeated reads faster?" In this example, Redis is used mainly for read caching, not as the primary database.

What Redis is Redis is an in-memory data store. In plain English, that means it keeps data in RAM so it can be read much faster than a full database query.
What "in-memory" means RAM is very fast. Because Redis keeps its values in memory, it is a good place to store temporary data that the app needs to read quickly.
What a cache is A cache is a temporary fast copy of data. The app checks the cache first so it does not have to rebuild the same answer every time.
What "source of truth" means The SQL tables User, Post, Comment, and Like still own the real data. Redis helps with speed, but it should not be the only copy of beginner-facing blog data in this example.
Two useful Redis words

Cache hit: Redis already has the value the app needs.

Cache miss: Redis does not have it yet, so the app must read from SQL, build the answer, and store a copy in Redis for next time.

How Redis Fits This Blog

Redis is best used here for repeated read results that many visitors ask for again and again.

Cache post pages by post_id A full post page or JSON payload is a strong caching candidate because the same popular post may be requested many times.
Cache comment counts by post_id Comment totals are simple summary values that are read often and quick to store.
Cache like counts by post_id Like totals are another common summary value that works well as a Redis key.
Optional: cache recent or popular lists A homepage list of recent or popular post IDs is another sensible Redis use case because many readers may request the same list.

Request Flow: App, Redis, and SQL

Why this is a different diagram from the ERD This picture shows system flow. It explains how the app checks Redis before SQL. It is not a table relationship diagram and should not be read like PK/FK links.

Worked Example: Reading One Popular Post

Step 1: A reader opens a post The app first checks Redis to see whether a cached post payload already exists for that post_id.
Step 2: Cache hit If Redis already has the post page, comment count, and like count, the app can respond very quickly.
Step 3: Cache miss If Redis does not have it, the app reads from the SQL tables, builds the result, then stores a temporary copy in Redis with a TTL.
Step 4: Later readers benefit The next visitor can be served from Redis instead of rebuilding the same result from scratch.

When Data Changes

Post edit If a post title or body changes, the app should delete or refresh the cached post payload for that post_id.
New comment If a new comment is added, the app should refresh or invalidate the post page cache and the comment-count cache.
New like If a new like is added, the app should refresh or invalidate the like-count cache for that post.
Default strategy A simple beginner-friendly approach is TTL plus targeted invalidation when the underlying SQL data changes.

Redis Data Structures for This Example

Suggested Redis Keys

Redis type Example key What it stores
String post:101:page Rendered post page or JSON payload for one post
String post:101:comment_count The current comment total for that post
String post:101:like_count The current like total for that post
Sorted Set posts:popular Popular or recent post IDs ordered by score

Why this matters: these keys model cache design, not SQL entities. They belong in the Redis layer, not the ERD.

Important beginner warning: caches can become stale If a post is edited, a new comment is added, or a new like appears, the cached Redis value may be out of date. That is why systems often refresh or delete cache entries when the main database changes.

What This Page Is Recommending

  • Cache full Post reads aggressively.
  • Cache Comment counts and Like counts strongly.
  • Teach full comment-thread caching as optional and selective.
  • Teach full raw like-list caching even more cautiously than counts.

Quick Quiz

Question 1 (Tables & Fields)

Which table stores who wrote a comment?

Answer: Comment, using user_id.

Question 2 (Tables & Fields)

What does the created_at field store?

Answer: A timestamp showing when the row was created.

Question 3 (Tables & Fields)

Which table would you query to find all comments on a specific post?

Answer: The Comment table, filtered by post_id.

Question 4 (Primary Keys & Uniqueness)

Can two users have the same user_id?

Answer: No - user_id is a primary key, so each value must be unique.

Question 5 (Primary Keys & Uniqueness)

Why is email marked UNIQUE?

Answer: So two users cannot share the same email address.

Question 6 (Foreign Keys)

What field links a post back to its author?

Answer: Post.user_id.

Question 7 (Foreign Keys)

The Comment table has two foreign keys. What are they and what do they point to?

Answer: post_id points to Post (which post), and user_id points to User (who wrote it).

Question 8 (Relationships)

What type of relationship exists between User and Post?

Answer: One-to-many (one user can author many posts).

Question 9 (Composite Keys)

Why does Like need both user_id and post_id?

Answer: Because a like is defined by one user liking one post.

Question 10 (Composite Keys)

Why is the Like table separate instead of being a field in the Post table?

Answer: Because many users can like many posts - a separate table handles this many-to-many relationship.

Question 11 (Data Integrity)

If Alice deletes her account, what happens to her posts?

Answer: It depends on the foreign-key rule the database uses. A system might block the delete, delete the posts too, or set the link differently.

Question 12 (Queries & Joins)

If you wanted to display all posts by a specific author, which two tables would you join?

Answer: User and Post, joined on User.user_id = Post.user_id.

Question 13 (Redis Basics)

What does "in-memory" mean when talking about Redis?

Answer: It means Redis keeps data in RAM so it can be read very quickly.

Question 14 (Redis Basics)

What is a cache hit?

Answer: Redis already has the value the app needs, so the app can return it immediately.

Question 15 (Redis Basics)

Which is the source of truth in this blog example: SQL or Redis?

Answer: SQL is the source of truth. Redis only stores temporary fast copies.

Question 16 (Redis Design)

What Redis key could store the comment total for post 101?

Answer: Something like post:101:comment_count.

Question 17 (Redis Design)

What should the app do if a new like makes a cached like count stale?

Answer: Refresh or invalidate the affected Redis key so the next read is correct.

Question 18 (Diagram Reading)

Why is Redis not drawn as a box in the ERD?

Answer: Because Redis is a cache layer and key design concern, not a relational entity with PK/FK relationships.
